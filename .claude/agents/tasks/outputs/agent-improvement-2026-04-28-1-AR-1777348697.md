**File:** `.claude/agents/pm.md`

### G3 — Critic Prescriptive Code Recipes in Complex Domains

**File:** `.claude/agents/critic.md`  
**Version bump:** 1.4.0 → 1.4.1 (PATCH)  
**Change:** Added "Recipe vs. problem-naming" guidance to §AUDIT_RISK close

**Source evidence:** gander-studio-p4 post-mortem §6 G3
- CR#1 wrote prescriptive recipe for A4 ("reset counter, assert === 2") without tracing useLinkSound.ts
- Recipe contained arithmetic flaw: `playApproach` also fires during drag (not just `playLink`)
- PM#2 implemented recipe faithfully
- CR#2 caught flaw via source trace after wasted PM round + CR round
- Cost: 1 PM round + 1 CR round + human authorization for 3rd Critic round (cap exceeded)

**Fix text:**
> In complex domains — audio synthesis, test spies, scheduler/event-loop logic, React Flow internals, build-tool internals — prefer naming the problem and pointing at the file over writing a prescriptive code recipe. Reserve prescriptive recipes for domains the Critic has fully traced and verified end-to-end via static reading.

**Mechanism:** Guidance insertion, enforced at Critic spec-read time.

**Effectiveness to monitor:**
- Watch next 2 sprints to see if new guidance changes Critic behavior
- If no change observed, escalate to hard gate rule: "BLOCK on prescriptive recipe in [domain matching X]"
