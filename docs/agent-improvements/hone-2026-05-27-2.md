### Confirmed scope (human refinement, this session)

The skill scope was discussed and expanded with the human before confirmation. The created skill should cover **rest-state text-on-background readability** with four sub-checks in a single Playwright pass:

1. **WCAG luminance contrast** — 4.5:1 for normal text, 3:1 for large (≥18pt or ≥14pt bold), threshold picked from computed `font-size` + `font-weight`.
2. **Alpha/transparency-aware background resolution** — when the element's own `background-color` is `rgba(*, 0)` or `transparent`, walk up the DOM to find the first non-transparent ancestor and use that as the effective background. (The S2 t6b bug hinged precisely on this — the textarea inherited `bg-transparent` from the Shadcn primitive and a naïve `getComputedStyle().backgroundColor` returns `rgba(0,0,0,0)`, producing a false PASS.)
3. **Color-vision-deficiency (CVD) simulation** — apply standard matrices for protanopia, deuteranopia, and tritanopia; recompute the ratio on each. Threshold: original colors must hit AA at 4.5:1, post-CVD-transformed colors must hit AA at 3:1 (a relaxed margin so the original palette has to be wider to survive CVD).
4. **Font-size-aware threshold pick** — derived from #1; called out as a discrete check because it changes the pass/fail decision and is easy to forget.

**Explicitly out of scope** for this skill (deferred to possible sibling or audit work):

- Interactive state contrasts: focus indicators (WCAG 2.4.7/11), hover, disabled, placeholder, `::selection` — these require triggering state, not just reading rest-state computed styles.
- Semantic color-meaning checks ("information conveyed by color alone", WCAG 1.4.1) — structural/design-system audit territory, not a computed-style smoke.
