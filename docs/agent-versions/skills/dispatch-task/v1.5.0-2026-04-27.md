# Dispatch Task — Full Delivery Cycle

This skill orchestrates one complete delivery cycle for a feature or task. It is invoked by the **Orchestrator** after receiving a request from the human. It wraps the PM → Critic → Execution → Audit → Archive sequence into a single, structured procedure.

## When To Use

Invoke this skill when:
- The human has a feature or fix to deliver and wants the full pipeline to run
- The task spans multiple domains (e.g., needs DS schema + BE endpoint + FE component)
- You (the Orchestrator) want the full team invoked in the correct sequence without manual step-by-step management

Don't use this skill for:
- Exploratory work or design discussions (use individual agents)
- Audit-only requests (use the audit-pipeline skill directly)
- Research tasks (spawn the researcher agent directly)

## Step 0: Establish Task ID and Design System Check

For sprints that include FE or UI work, after establishing the task_id, check whether `DESIGN.md` exists at the target app root:

```
Glob: {APP_ROOT}/DESIGN.md
```

- **Found:** note its path. The PM will include it in FE/UI task packet `<context_files>`.
- **Absent:** surface this to the human before proceeding:
  > "No DESIGN.md found at [app path]. The UI Designer and Frontend Engineer will operate on inferred tokens this sprint, which may produce inconsistent visual output. Recommend running `/generate-design` first. Reply 'proceed anyway' to skip, or 'run design first' to bootstrap the design system before decomposition."

  If the human says proceed anyway, flag it in the orchestrator_brief to PM. If they run design first, invoke the `generate-design` skill, wait for confirmation, then continue.

This check is non-blocking — it costs one glob call and surfaces a known risk. Skip it only for sprints with zero FE work.

## Step 0 (cont.): Establish Task ID

Before anything else, the Orchestrator defines a `task_id` for this delivery cycle. Every file output path, event log entry, SPAWN event, and agent handoff uses this ID. Without it, agent default output paths contain a literal `{task_id}` string and the event graph cannot be reconstructed.

Convention: `{project}-p{N}-{slug}` — e.g. `poster-p1-vector-base`, `poster-p3-palettes`

Write the task_id into:
- Every `<orchestrator_brief>` and `<task_packet>` dispatched to an agent
- Every SPAWN event logged to `docs/events/agent-events-{YYYY-MM-DD}.jsonl`
- The `## Output Path` block included in every agent prompt

## Step 0.4: Working-Tree Cleanliness Check

**Required for any sprint whose success criteria include `git diff` evaluation against the prior commit** (cumulative-diff SCs — e.g. SC4 patterns of the form `git diff HEAD -- {paths} | grep '^[-]…' | wc -l == 0`). Skip only when no SC in the planned task packets references `git diff` against HEAD or a named commit.

Before briefing the PM, run:

```bash
git -C {project_root} status --porcelain -- {target_paths}
```

`{target_paths}` is the set of files or globs the sprint is scoped to touch. For catalog-wide sprints this is the catalog root (e.g., `.claude/skills/`). For single-file sprints it is the file.

**If the output is empty:** proceed.

**If the output shows pre-existing modifications to any target path:** halt and surface to the human:

> "Sprint `{task_id}` proposes diff-gated SCs against {paths}. The working tree shows pre-existing modifications to {N} target files from prior sprints (likely `{prior_sprint_id}`). A `git diff HEAD` will conflate this sprint's changes with the inherited delta, which can cause the auditor to flag scope-creep on lines this sprint did not touch. Options: (a) commit the inherited modifications to a separate sprint first, (b) explicitly scope the SCs around the pre-existing changes (record in PM brief `<context>`), or (c) acknowledge the risk and proceed. Reply with `commit-first`, `scope-around`, or `proceed-anyway`."

Do not proceed past this step without an explicit human response. Log the chosen option in the event log:

```json
{"seq":{N},"ts":"{ISO-8601}","ev":"WORKTREE_CHECK","task_id":"{task_id}","agent_id":"ORC#0","status":"{clean|inherited_modifications}","resolution":"{none|commit-first|scope-around|proceed-anyway}"}
```

(Post-mortem root cause: `gander-p2-skills-when-to-use` §6 G1 — HR#1 inherited uncommitted state from `hone-2026-04-23-2` and `agent-improvement-2026-04-23-1` on `agent-improvement/SKILL.md`; the SC4 cumulative-diff check flagged the inherited delta as if HR#1 had produced it; one full audit-fail remediation round was burned reverting work HR#1 had not actually done.)

## Step 0.6: PM Context Preflight (Agent-Spec Constraint Extraction)

**Required for any sprint that edits, creates, or routes work to specific agent specs.** The PM repeatedly applies plausible-but-wrong defaults to live agent specs — citing remits, write authorities, or constraint sections that contradict what's actually written in the spec. Three sprints have surfaced this same failure class: `gander-p2-hone-skill` (PM didn't read source files), `gander-p3-team-report-v1.2` (PM emitted stub output), and `gander-p4-dashboard-language` (PM cited an HR remit that excluded `.claude/refs/`). The Critic catches it consistently — but the Critic is doing PM's job.

This step removes the PM's read-and-remember burden by pre-extracting the binding constraint text from every agent spec the sprint targets and inserting it verbatim into the PM's `<orchestrator_brief>` `<context>` field. The PM no longer infers an agent's remit; the remit arrives quoted from the live spec.

**Trigger:** any of these hold —
- The sprint will route any task packet to an agent (every full-pipeline sprint).
- The sprint will edit `.claude/agents/*.md` or `.claude/skills/**/*.md` (HR-targeted sprints).
- The human request mentions a specific agent by name ("HR will...", "the auditor needs...").

**Skip when:** the sprint is data-terminal (RA-only or ST-only, no PM decomposition), or the request is a remediation loop after audit FAIL on a single file.

**Procedure:**

1. Identify the **target agent set** for this sprint — every agent whose spec will be edited, every agent whose output is referenced as a contract source, and every agent that will receive a task packet. Default set when uncertain: `pm`, `hr`, plus any agent code mentioned in the human's request.

2. For each agent in the target set, run this extraction against `.claude/agents/{name}.md`:

   ```bash
   awk '
     /^## (What .+ Can Modify|What .+ Does Not Modify|Constraints|Boundaries|Write Authority|Remit|Scope)/ {capture=1; print; next}
     capture && /^## / {capture=0}
     capture {print}
   ' .claude/agents/{name}.md
   ```

   The extracted block is the **binding constraint text** for that agent. If the awk returns nothing, surface to the human — the agent's spec lacks an explicit remit section, which is itself a finding.

3. Concatenate every extracted block into a single `<agent_remits>` XML structure:

   ```xml
   <agent_remits source="auto-extracted from .claude/agents/*.md at {ISO-8601}">
     <agent name="hr" version="{frontmatter version}">
   {verbatim extracted text}
     </agent>
     <agent name="auditor" version="{frontmatter version}">
   {verbatim extracted text}
     </agent>
     <!-- one per target agent -->
   </agent_remits>
   ```

4. Insert the `<agent_remits>` block into the PM's `<orchestrator_brief>` under `<context>`. The PM brief MUST include this preamble before the `<task>` block:

   > The `<agent_remits>` block below is auto-extracted verbatim from the live agent specs at the timestamp shown. Treat its content as authoritative. Do NOT infer an agent's remit, write authority, or constraint set from prior sprints, training-data assumptions, or this sprint's narrative — quote the `<agent_remits>` block directly. If a task packet's routing requires an agent to act outside its quoted remit, surface that as a precursor `<task_packet>` (e.g., "extend HR's write authority to cover X") rather than asserting the remit covers it implicitly.

5. Log the extraction as an event:

   ```json
   {"seq":N,"ts":"{ISO-8601}","ev":"PM_PREFLIGHT","task_id":"{task_id}","agent_id":"ORC#0","agents_extracted":["hr","auditor",...],"out_path":"docs/task-registry.md#agent_remits"}
   ```

   Mirror the `<agent_remits>` block into `docs/task-registry.md` under a `## Agent Remits` heading so post-mortem reconstruction can verify what the PM was given.

**Deterministic extraction, not judgment.** This step is mechanizable — `awk` reads section headings and prints their bodies. The Orchestrator runs it directly; no agent is spawned. (Code-not-prompt rationale: the failure is "PM applied plausible default that contradicted live spec text"; the fix is "PM doesn't get to apply defaults because the spec text is in front of it." Removing the judgment moment removes the failure mode.)

(Post-mortem root cause: `gander-p4-dashboard-language` §6 Gap 2 — third recurrence of PM citing agent constraints that contradict the live spec. Critic remediation rounds for this class of error are now structurally avoidable.)

## Step 0.7: Scry (Optional)

Before briefing the PM, the Orchestrator may invoke the `scry` skill to run an
evidence-gathering round. RA produces an `<evidence_brief>` (structured XML with per-claim
provenance) that the PM consumes as part of its `<orchestrator_brief>` context. This grounds
the PM's decomposition in cited external facts rather than training-data assumptions.

**Invoke scry when any of these hold:**
- The human's request references a third-party API, SDK, or dataset that may have changed.
- A prior sprint for the same domain had a Critic flag of MISSING_RESEARCH.
- The Orchestrator's own intake review flags external-dependency uncertainty.
- The human explicitly says "research first" or "verify the docs."

**Skip scry when any of these hold:**
- The sprint is purely intra-codebase (no external dependencies).
- RA ran for the same external system < 30 days ago and findings are still fresh.
- This is a remediation loop after an audit FAIL.
- The sprint is data-terminal (RA or ST is the executor — no PM decomposition follows).

**If invoked:**
1. Log a SPAWN event for RA with `"edge_label":"evidence_brief"`.
2. Include in RA's prompt: the research question (what facts the PM needs), the sprint slug, and the ## Output Path block.
3. RA returns an `<evidence_brief>` XML block.
4. Include the full `<evidence_brief>` block verbatim in the `<context>` field of the PM's `<orchestrator_brief>`.
5. PM's task packets may reference findings by `finding id` — e.g. "per evidence_brief finding #2, the API returns lat/lon as strings."

**Cap:** two RA rounds per sprint. A `<credential_request>` from RA surfaces to human and blocks the sprint on that dependency.

Full skill procedure: `.claude/skills/scry/SKILL.md`.

## Step 0.8: Name Confirmation (Naming-Novel Sprints)

If the sprint introduces a novel top-level name — a new skill directory, a new agent file, a new command, a new top-level concept the human has not yet fixed in writing — confirm the name with the human **before** briefing the PM. A mid-sprint rename after HR/implementer output has landed costs an `sed` sweep across all task outputs, task packets, changelog entries, and the skill/agent file tree, and requires bespoke translation instructions to later waves that reference the PM plan by old name.

**Trigger:** any of these hold —
- The plan will create a new `.claude/skills/{name}/` directory.
- The plan will create a new `.claude/agents/{name}.md`.
- The plan will add a new `/command` to `.claude/commands/`.
- The human's request describes a new top-level concept by a working name ("some sort of cleanup skill", "a preflight research thing") rather than a fixed name.

**Procedure:** surface the proposed names to the human in a single message:

> "This sprint introduces {N} novel names: {list}. Confirm these names before decomposition, or propose alternatives. A rename after agents start running costs a full sed sweep of task outputs and the changelog."

Wait for confirmation. Record the confirmed names in `docs/task-registry.md` under a `## Names Registry` heading (populated in depth by `assign-agents` Step 0.5). Then proceed to Step 1.

**Skip when:** the sprint modifies only existing agents/skills, adds no new top-level name, or the human has already fixed the names in the request itself. (Post-mortem root cause: `gander-p2-skills-ra-cleanup` — PM#1 invented names `ra-preflight` and `hygiene`; human mid-sprint renamed to `scry` and `tidy-up`; ORC ran `sed` across 5 files + HR#1 output packet with 35 replacements and gave HR#2 a bespoke translation brief.)

## The Cycle

### Step 1: PM Decomposition

The Orchestrator spawns the PM agent with an `<orchestrator_brief>`. The PM will:
- Break the request into atomic `<task_packet>` assignments
- Identify domain dependencies (DS before BE, BE before FE live wiring)
- Assign priority levels
- Return a `<task_decomposition>` block

**Review the task decomposition before proceeding.** If the decomposition is wrong (wrong agents assigned, missing dependencies, misunderstood scope), send a `<revision_request>` back to PM. Don't execute a flawed plan.

### Step 1.5: Critic Review (Plan Gate)

Before spawning any implementing agent, spawn the Critic with the PM's full `<sprint_state>` and all `<task_packet>` blocks.

Log a SPAWN event first:
```json
{"seq":{N},"ts":"{ISO-8601}","ev":"SPAWN","task_id":"{task_id}","agent_id":"CR#1","parent_id":"MAIN","edge_label":"plan_critique","expected_output":".claude/agents/tasks/outputs/{task_id}-CR-{unix_ts}.md"}
```

Include in the Critic's prompt:
- The full `<sprint_state>` from PM
- All `<task_packet>` blocks
- The original human request (verbatim)
- The `## Output Path` block pointing to `.claude/agents/tasks/outputs/{task_id}-CR-{unix_ts}.md`

**If the Critic returns `CRITIQUE_BLOCK`:**
- Do not spawn any implementing agent
- Return the full `<plan_critique>` to the PM as a `<revision_request>`
- PM must address every BLOCKER challenge and produce a revised `<sprint_state>`
- Re-run Step 1.5 on the revised plan
- If the same blocker appears in two consecutive CRITIQUE_BLOCK results → escalate to human with a summary of both critique outputs

**If the Critic returns `CRITIQUE_PASS` (clean — no WARNINGs):**
- Evaluate Step 1.6 below (Jidoka) — opt-in; skip by default for simple sprints
- Proceed to Step 2 unless Jidoka is invoked and surfaces a re-partition

**If the Critic returns `CRITIQUE_PASS` with WARNING-level items (warning-resolution branch):**
- Do NOT proceed to Step 2 yet. WARNINGs are not cosmetic — an unacted WARNING is a predictable audit snag the Critic already identified.
- Forward the full `<plan_critique>` to PM as a `<warning_resolution_request>`. PM must return a compact `<plan_amendment>` that lists only the specific SC tightenings, `<out_of_scope>` additions, or risk acknowledgements addressing each WARNING.
- Do NOT re-decompose, do NOT re-run `<task_decomposition>`, do NOT re-invoke the Critic. Warning-resolution rounds are explicitly **outside the two-round Critic cap and outside the two-round jidoka cap** per `.claude/agents/orchestrator.md` §Warning-resolution round scope (the authoritative rule).
- If PM's amendment exceeds SC-level changes (adds/removes task_packets, reorders dependencies, renames tasks), treat it as a BLOCK-class revision and re-enter Step 1.5 Critic gate.
- On receipt of an SC-level `<plan_amendment>`, log a `plan_amendment` SPAWN with a `note` field citing `warning_resolution_request - outside Critic/jidoka round caps per orchestrator.md v1.7.2` so post-mortem reconstruction can distinguish a warning-resolution spawn from a BLOCK-driven re-plan.
- Then evaluate Step 1.6 (Jidoka) and proceed to Step 2 as in the clean branch.

**If the Critic returns `CRITIQUE_BLOCK`:** (see block above)

The Critic does not slow down clean plans — a PASS costs one agent turn. It stops broken plans before they consume four times that in remediation loops. The warning-resolution branch is a third-state middle path: neither a BLOCK (no full re-plan) nor a clean PASS (no silent carry-through of predicted audit snags). (Post-mortem root cause: `gander-p2-new-skills-log-commit` §8c — this skill previously documented only two Critic outcomes, with WARNINGs described as "surface to human as a brief note"; the actual protocol in orchestrator.md §Warning-resolution round scope routes them to PM for amendment. ORC executed the correct protocol de facto, but the skill's written procedure did not match.)

### Step 1.6: Jidoka Planning Round (Optional)

Between Critic PASS and `assign-agents`, the Orchestrator may invoke the `jidoka` skill to run a plan-only preflight round. Each implementing agent assigned in the approved plan is dispatched in plan-only mode, reads its actual source files, and returns a `<plan_packet>`. The Orchestrator synthesizes the plan_packets; if they surface a re-partition need or a fact the plan assumed wrongly, the PM re-decomposes and the revised plan re-runs Step 1.5 (Critic) before this step or Step 2 runs again.

**Invoke jidoka when any of these hold:**
- The plan has 4+ implementing task packets with aggregate context_files ≥15.
- Any task packet's estimated_new_lines exceeds 150 or is expected to touch 4+ files in one domain.
- The Critic's `<plan_critique>` emitted DEPENDENCY or SCOPE WARNINGs citing PM assumptions about the codebase.
- The PM's `<routing_notes>` contain `jidoka_recommended: true`.

**Skip jidoka when any of these hold:**
- Sprint has a single implementing task.
- All context_files are already embedded in the PM brief.
- The Critic returned zero fact-level warnings.
- The sprint is data-terminal (RA or ST is the executor).
- This is a remediation loop (targeted single-file fix after audit FAIL).

Full procedure, output contracts, and security rules: `.claude/skills/jidoka/SKILL.md`.

**After jidoka:**
- `JIDOKA_PASS` → proceed to Step 2 with the original plan.
- `JIDOKA_REPARTITION` or `JIDOKA_REPLAN` → route `<jidoka_synthesis>` to PM as `<jidoka_revision_request>`. PM produces revised `<task_decomposition>`. The revised plan re-runs Step 1.5 (Critic) — **jidoka never bypasses the Critic gate.** If Critic re-passes, evaluate jidoka again (one retry max per sprint); otherwise proceed to Step 2.

### Step 2: Execute via assign-agents skill

Invoke the `assign-agents` skill. It handles:
1. Building the full assignment registry (brief, context, scope boundaries, expected return per agent)
2. Deriving the spawn wave order from declared dependencies
3. Logging all SPAWN events before dispatching
4. Writing the expectation manifest to `docs/task-registry.md`
5. Spawning Wave 0 agents with complete, unambiguous briefs
6. Receipt-checking each return packet against the manifest before handing to the auditor
7. Spawning subsequent waves as earlier waves complete

Do not spawn implementing agents manually — use the skill. It enforces the brief format, the SPAWN event log, and the receipt check that would otherwise be skipped under execution pressure.

**Hard gate — expectation_manifest precondition for Step 3.** Before invoking the auditor on any returning packet, verify `docs/task-registry.md` contains an `<expectation_manifest>` block whose `<sprint_id>` matches the current task_id prefix. If it does not, `assign-agents` was bypassed — halt, log an `ASSIGN_AGENTS_BYPASS` observation to the event log, and re-enter Step 2 to produce the manifest before any audit runs. Do not let a missing manifest pass because "the content is fine" — the manifest is the schema the receipt check reads from, and a sprint that audits without one cannot be validated post-hoc. (Post-mortem root cause: `gander-p2-hone-skill` §6 Gap 3 — ORC wrote Names Registry + Expectation Manifest inline, bypassing `assign-agents`; enforcement depended on ORC discipline alone.)

### Step 2.5: Receipt-Check Gap Routing

The receipt check inside Step 2 (`assign-agents` duty 6) compares each returning packet against the `<receipt_check>` items in the `<expectation_manifest>`. It runs **before** Step 3 audit dispatch. Two routing outcomes are possible when an item does not pass:

- **`gap_request` (light path).** Route when ALL of the following are true: (a) the failing item corresponds to a single, mechanically-defined success criterion (one grep/awk/wc check, not a judgment call); (b) the fix is a bounded single-file, single-token or single-line substitution with no downstream consequences (no schema drift, no cross-file impact, no semantic change to the deliverable); (c) the violation was not caught by the implementing agent's self-verification before emission (i.e., a formatting oversight, not a misunderstanding of scope). Spawn a fresh implementer (same agent code, next instance number — e.g. HR#3 → HR#4) with a compact `<gap_request>` brief naming the specific SC, the offending line, and the required substitution. On return, re-run the receipt check; if it passes, proceed to Step 3. The full auditor then sees the fixed file on first audit, not a remediation-round artifact.

- **Full audit dispatch (normal path).** Route when the failing item is multi-faceted, requires judgment, or the fix touches multiple files or semantics. Dispatch the auditor against the packet as-is; the auditor issues a `<remediation_request>` and Step 3's `ralph-loop` drives the remediation cycle.

**Decision rule in one sentence:** if the fix is a `sed`-able one-liner bound to one SC, route as `gap_request`; otherwise, dispatch the full audit and let the auditor's remediate cycle handle it.

Log `gap_request` spawns to the event log as a distinct SPAWN (with a `note` field describing the gap, e.g. `"receipt_check gap: SC7 XML-tag-in-description failure"`) so post-mortem reconstruction can tell a gap-fix cycle from an audit remediate cycle. Do not undercount: a `gap_request` is a separate implementer spawn and must appear in agent-activity counts, but it is NOT an `AUDIT_FAIL` and must not be tallied as one.

(Post-mortem root cause: `gander-p2-new-skills-log-commit` §6 Gap 2 — receipt-check caught SC7 XML-tag-in-description on HR#3's commit-packet delivery; ORC routed as `gap_request` to HR#4 (4-byte `<id>` → `{id}` substitution) before audit, saving a full AUDIT_FAIL → remediate → re-audit cycle. Pattern was exercised but undocumented; codified here.)

### Step 3: Audit Each Completion

After each implementing agent returns its packet, invoke the audit-pipeline skill. Do not wait until all agents finish — audit each domain's output as it arrives.

If an audit fails:
- Return the `<remediation_request>` to the implementing agent immediately
- Use the ralph-loop skill to drive the remediation cycle
- Do not start the next dependent task until the failing task passes audit

### Step 3.5: Requirements Validation

After all tasks in the sprint (or wave, for multi-wave sprints) have passed audit, invoke the `requirements-validate` skill **before archiving anything, every time**, regardless of how trivial coverage appears. The skill reads the original `orchestrator_brief`, all task packet success criteria, and the delivered output files, and produces a `<requirements_coverage_report>`.

**If overall_status is COVERED:** proceed to Step 4.
**If PARTIAL or MISSING:** do not archive; route each gap as a `<gap_fill_request>` (BLOCKER priority) to the implementing agent; re-run pipeline for the gap; the sprint does not close until every requirement is COVERED.

**No shortcut.** Inline ORC self-verification — even for a 2-item sprint — sets a precedent that erodes at scale. Always invoke the skill. (Post-mortem root cause: `gander-p2-skills-ra-cleanup` ORC self-verified coverage inline before archiving because the mapping was trivial; acceptable for that sprint but a slippery-slope precedent.)

**Artifact hard gate.** The skill must produce a `<requirements_coverage_report>` file at `.claude/agents/tasks/outputs/{task_id}-REQVAL-{unix_ts}.md`. Before proceeding to Step 4, verify the file exists on disk and its `overall_status` parses as one of `COVERED | PARTIAL | MISSING`. If the canonical artifact is not present, the skill was not invoked — re-enter Step 3.5 and run it as a skill. Inline synthesis in the main session does not produce this artifact and therefore cannot satisfy the gate. (Post-mortem root cause: `gander-p2-hone-skill` §6 Gap 2 — ORC executed the procedure inline; output file was written but formal skill invocation was bypassed, same anti-pattern as the prior sprint despite the "no shortcut" rule.)

### Step 3.6: Durability Commit (commit-packet)

After REQVAL returns COVERED and **before** the Archivist runs, invoke the `commit-packet` skill on each audit-passed packet. This step exists as a discrete entry to make the close-out chain ordering explicit:

```
audit PASS → REQVAL COVERED → commit-packet (this step) → Archivist
```

The Archivist's archive_entry should reference an actual git commit hash with a `task: {task_id}` trailer; that is only possible if commit-packet ran first. Inverting the order — Archivist before commit-packet — produces an archive_entry containing "Commit pending" claims that go stale when the commit subsequently lands. (Post-mortem root cause: `gander-p3-team-report-v1.2` §6 G6 — two parallel ORCs read the implicit closeout ordering differently and produced two different sequences; making it explicit here removes the ambiguity.)

**Per-packet rule.** Each audit-passed packet gets its own commit. Do not batch multiple packets into one commit unless they share the same task_id and the same files (e.g., serialized HR#1 + HR#2 edits to a single shared file where the diffs are intermingled in the working tree post-hoc).

**Artifact hard gate.** The `commit-packet` skill must produce a `<commit_manifest>` file at `.claude/agents/tasks/outputs/{task_id}-COMMIT-{unix_ts}.md` containing one `<commit>` block per packet committed in this wave. Before proceeding to Step 4, verify the file exists on disk and that its `<commit>` count equals the number of audit-passed packets dispatched to commit-packet. If the canonical artifact is absent or under-counted, the skill was not invoked (or was bypassed mid-wave) — re-enter Step 3.6 and run it as a skill. Inline `git commit` invocations from the main session do not produce this artifact and therefore cannot satisfy the gate. (Post-mortem root cause: `gander-p4-dashboard-language` §6 Gap 3 — ORC executed 8 sequential commits inline; output was correct but no commit-packet skill invocation occurred, same anti-pattern flagged across audit-pipeline, requirements-validate, and assign-agents in prior sprints.)

### Step 4: Archive Completions

After each task passes audit *and* commit-packet has produced a commit (Step 3.6), spawn the Archivist to log the completion with rationale. Don't batch archive entries — log each one when it closes. The Archivist verifies the commit landed by grep'ing `git log -1 --format=%B HEAD` for `task: {task_id}` before asserting commit completion in the entry (see `archivist.md` Commit-precondition).

### Step 4.5: Human Verification Checkpoint (FE tasks only)

For any sprint that includes FE work, **before declaring the sprint DONE**, request the human to do a manual browser check. The Playwright smoke test catches JS runtime errors, but it cannot catch visual regressions, map tile rendering failures, or interaction bugs that require a real user.

Post this to the human:

> "All audits passed. Before I close the sprint, please open the app in your browser and confirm: (1) the page loads, (2) the new feature is visible and behaves as expected, (3) the browser console shows no errors. Reply with OK to close the sprint, or describe any issue found."

**Do not mark the sprint DONE until the human confirms.** If a runtime issue is found, open a new remediation task with priority BLOCKER and run it through the full pipeline (implement → audit → archive) before closing the sprint.

This step exists because bundler misconfigurations, tile rendering failures, and interaction bugs only surface in a live browser — not in static analysis or headless Playwright.

### Step 5: Report to Human

When all tasks in the sprint are DONE and human verification has passed, produce a final `<sprint_state>` showing:
- All task IDs completed
- Any tasks that were modified in scope during delivery (and why)
- Any open risks or follow-up tasks identified during implementation
- Key files changed (for the human's awareness)

## Parallel vs. Sequential Execution

```
Example: "Add user profile editing"

Sequential (dependency chain):
  DS: add profile fields migration →
  BE: add PATCH /users/:id endpoint + Zod schema →
  FE: build profile edit form + wire to API

Parallel (independent):
  BE: user endpoint     ┐
  BE: settings endpoint ┘ → both audited independently
```

When tasks are independent, spawn them simultaneously. When there's a dependency (BE needs DS migration first), enforce the sequence.

## Communication Overhead

This skill reduces communication overhead by making the routing explicit and automatic. The human's involvement is:
1. Provide the goal to the Orchestrator
2. Review the Orchestrator's summary of the PM's task plan (optional but recommended)
3. Confirm browser verification for FE tasks (Step 4.5)
4. Receive the Orchestrator's final plain-language delivery report

The human should not need to say "now audit this" or "send it to the archivist" — the Orchestrator handles that via this skill.

## Post-Mortem (Optional, Recommended for Sprints with Failures)

After the sprint is fully closed and the human has confirmed (Step 4.5), invoke the `post-mortem` skill if:
- Any audit produced a FAIL result during the sprint, **or**
- A runtime bug was discovered post-delivery, **or**
- The human explicitly requests a retrospective

The post-mortem skill gathers all artifacts produced during the sprint — event log, agent outputs, audit results, archivist entries — and synthesizes them into a structured retrospective at `docs/post-mortems/{sprint-slug}.md`. The archivist logs it to the project log.

For clean sprints (zero audit failures, no post-delivery bugs), a post-mortem is optional. The archivist's `archive_entry` records are sufficient.

## Escalation

This skill respects all existing escalation rules:
- 3 consecutive audit failures → escalate to human
- 10 ralph-loop iterations without resolution → escalate to human
- PM identifies a dependency or requirement that wasn't in the original request → pause and check with human before continuing
