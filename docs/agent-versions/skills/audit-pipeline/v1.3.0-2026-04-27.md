# Audit Pipeline

## How This Skill Is Invoked

This skill is a **procedure the Orchestrator calls, not a recipe it transcribes**. Spawning the `code-auditor` agent directly from the main session with a verbose custom prompt produces the same output but bypasses the skill's packaging — the consistent prompt shape, the standard scope boundaries (SA → QA → SX sequence, meta-agent independence rule, pipeline integrity check in Step 2.5), and the standard audit-output path. Direct dispatch is a silent drift source: the agent runs, the review lands, but the protection the skill provides is lost.

Always invoke via the Skill tool (or let `dispatch-task` Step 3 trigger it) rather than dispatching the auditor directly. If you find yourself writing "I'll just spawn code-auditor with a custom prompt," stop — the custom prompt is the skill, and the skill is the custom prompt. Transcribing is drift. (Post-mortem `gander-p2-hone-skill` §8c: ORC dispatched `code-auditor` directly twice instead of invoking this skill; output was correct by luck, but the AMBIGUOUS_STEP root cause was the same as for `assign-agents` and `requirements-validate` in the same sprint.)

---

## When To Use

After any subagent returns output that includes **non-transient file writes**. This includes:

- Backend, frontend, and database implementation agents (original case)
- System-health / meta-agent work: HR (agent-team modifications), SK (skills management), configuration edits to `.claude/`, `settings.json`, or hooks
- Any other agent output that modifies files committed to the repository (agent/skill specs, rules, documentation with durable repo impact)

Skip only when the output is transient (task-outputs under `.claude/agents/tasks/outputs/`, scratch files). No task with durable file writes is done until this pipeline runs and passes.

(Post-mortem: `consolidate-agents-2026-04-16` — a 133k-token, 13-file system-health consolidation shipped with zero audit because the Orchestrator treated meta-agent work as exempt. The sprint's entire output was destroyed ~30 min after COMPLETE without anyone verifying what had been written.)

## Procedure

1. Spawn the auditor agent with:
   - The changed file paths (not the entire codebase — scope it)
   - The task's success criteria from the original `<task_packet>`
   - The agent's output packet (`completion_packet`, `ui_packet`, or `data_packet`)

2. The auditor runs three checks in sequence: **Standards → QA → Security**. Each check stops at the first FAIL. For FE tasks, QA includes a live Playwright browser smoke check (Tier 1 always; Tier 2 if a spec file exists) — the auditor manages dev server lifecycle automatically. The PM does not need to orchestrate this separately.

2.5. **Pipeline integrity check.** Before accepting the auditor's verdict as sprint-closing, scan the sprint's event log (`docs/events/agent-events-{YYYY-MM-DD}.jsonl`, entries matching the sprint's `task_id`). If every SPAWN/COMPLETE entry in the sprint shares the same base agent_id suffix — all `#0-direct`, or all the same `#n` under a single parent with no distinct child spawns — the sprint ran in single-agent (ORC-direct) mode. Emit `pipeline_integrity: WARNING` in the audit output and require explicit human acknowledgement before marking the sprint closed. An all-direct event log is diagnostic evidence that the Critic gate, the independent audit, and any HR review were structurally impossible. For meta-agent sprints (writes to `.claude/`, `~/.claude/`, agent/skill specs) this condition also triggers the Auditor's Meta-Agent Independence Rule — the verdict must be downgraded to INDETERMINATE. (Post-mortem root cause: `gander-p1-jidoka-skill` (2026-04-22) and `canvas-skill-p1` (2026-04-18) both produced all-`#0-direct` event logs; audit-pipeline passed them without flagging the degradation.)

2.7. **CI-status check (target projects with CI deployment).** If the target project ships via CI (GitHub Pages, Vercel, Netlify, GitHub Actions deploy workflow), a static-and-Playwright PASS is necessary but not sufficient. The audit does not advance to PASS until the latest CI run for the audited commit is `success`. Run:

```bash
gh run list --branch {branch} --limit 1 --json conclusion,workflowName,headSha,status
```

If `conclusion` is anything other than `success` (`failure`, `in_progress`, `cancelled`, `null`), the audit-pipeline does not PASS. Surface the failing workflow name and, if `gh run view --log-failed` reveals a specific cause, include the first error line in the audit output. A commit that lints clean and tests green locally but breaks the CI deployment (dangling symlink, missing artifact, environment mismatch, oversized artifact) is a failed shipment — the implementing agent's "shipped" claim must be conditioned on a green CI run for the audited commit, not on a clean static check.

Skip this step only when the project has no CI workflow (`gh workflow list` returns empty) or when the audited commit has not yet been pushed to the deploy branch — in the latter case, audit may PASS conditionally with a note that CI verification is pending push, and the Orchestrator's deployment-status check on the next "is X live" question will close the loop.

(Post-mortem root cause: `session-2026-04-27-pages-build-fix` §5 G2/G4 — `broadn-p8-feedback-widget` post-mortem claimed "shipped to GitHub Pages preview" because static QA passed. The widget never actually deployed: a dangling-symlink defect committed 4 days earlier broke every Pages build for the entire window between p8 close and human discovery. A CI-status check at audit time would have caught the defect on commit `b1ab566` instead of 4 days later via human-eyeball detection.)

3. **If all PASS:** Mark the task complete. Spawn the archivist to log the completion with rationale.

4. **If any FAIL:** Return the full auditor report to the implementing agent with a single, specific remediation request. The implementing agent must address the flagged issue — not work around it.

5. After remediation, re-run the full audit. This is non-negotiable — a partial re-check can miss cascading issues introduced during the fix.

6. **If the same issue causes 3 consecutive audit failures:** Escalate to the human with a summary of what was attempted and what failed each time.

## Failure Handling

When returning an audit failure to an implementing agent, format the remediation request clearly:

```xml
<remediation_request>
  <task_id>[ID]</task_id>
  <attempt_number>[1 | 2 | 3]</attempt_number>
  <failing_check>[SA | QA | SX]</failing_check>
  <specific_issue>[exact violation from the audit report — not a paraphrase]</specific_issue>
  <required_fix>[what the agent must change — be precise about file and line if known]</required_fix>
</remediation_request>
```

Passing vague feedback ("fix the security issue") produces vague fixes. Quote the auditor's finding directly.

## Timing

Run the audit pipeline *before* logging to the archivist. The archivist records verified, complete work — not work-in-progress. The sequence is always: implement → audit → archive.
